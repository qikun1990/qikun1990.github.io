    Day 18: Setting Up Hosting and CDN

    Now that we've ripped some landers, we'll need to find a place to serve them to online visitors - so we'll be setting up hosting in this lesson.

    We'll also need to register a domain name for your landers, and nameservers with fast redirection speed to allow the landers to start loading as early as possible.

    And last but not least, we'll need to set up a CDN service to serve your landers from locations across the globe, to make pages load fast for visitors from different countries. As you'll see in a future lesson, page load speed is ESSENTIAL especially when it comes to pop traffic, because nobody is going to stand around waiting for an intrusive ad to load.

    Please Note: We will be using Namecheap as the domain registrar, Route 53 as the DNS service provider, Amazon S3 as the webhost, and Amazon Cloudfront as the CDN. If you want to substitute any of these services, please feel free to do so, but know that if you run into problems, you'll be on your own (you can still ask questions in the forum and try to get other members to help - just that I may not be able to help you because I'm weak when it comes to tech).

    Lots of instructions to follow today so LET'S BEGIN!

    ACTION


    1)REGISTER A DOMAIN AT NAMECHEAP


    -Register an account on Namecheap.com if you don't yet have one.


    -Sign in, click on "Domains" > "Domain Name Search".


    -Think of a domain name for your landing pages. If you have a vertical you would like to focus on, you can come up with a domain name that is related to that vertical (e.g. for sweeps, something like daily-prize-rewards.com). If you haven't decided on what kind of offers to promote, then just register something generic, such as:

    exclusive-offers-for-you
    special-promotional-offers
    extreme-deals-site
    today-only-special-offers

    ....you get the idea. I wouldn't suggest spending too much time on trying to come up with the perfect domain name, because nowadays, domains get banned by google quite frequently.

    Enter your desired domain name into the search field and press ENTER. Domain names that have already been taken will say "TAKEN" and show a "make offer" icon at the end of the row. Domain names that are available will show an "add to cart" icon.



    And no, it doesn't have to have a .com extension. For example, .xyz, .club, and .life domains are cheaper. However, keep in mind 2 things:

    a)People tend to trust .com sites more - I haven't personally split-tested this myself though, so have no idea how much impact domain extensions can have on conversion rates.

    b)Some people believe that .com domains are less likely to be flagged/banned by google.

    You're encouraged to do your own experimentation. For your first domain, don't overthink - just get SOMETHING registered.


    -Once you've decided on a domain name, click on "add to cart", then click on "View Cart" where you'll need to complete the purchase.

    No need to buy their additional services:



    Purchasing for 1 year would be enough - you can always pay for another year if the domain isn't flagged by google by then. "AUTO-RENEW" I would suggest to leave at OFF for the same reason (although you could leave it on, and come back to turn it off if the domain gets flagged by google - your choice).

    Get WhoisGuard (now "free forever" on Namecheap), as many pop landers are at least somewhat scammy in nature, so it would be best to not have your name associated with those landers.

    No need to get Premium DNS - we'll be using Amazon Route 53, as you'll see later.



    I purchased a new domain for use with this tutorial as an example - the domain name is extremedeals.club. I'll be using this domain for the remainder of the tutorial.



    2)SET UP HOSTING on AMAZON S3


    This is the method taught by Caurmen in this post. Parts of this section are copied from that post (thanks Caurmen-the-Great!)


    -Register an account at AWS if you don't yet have one: https://aws.amazon.com/


    -Sign into the AWS Console. Click on "Services", then "S3" under the "Storage" section. Click on "+ Create bucket".


    -In the "Create bucket" window, put your domain name (can include the extension) as the "Bucket name", leave "Region" at default (location doesn't matter as we'll be using a CDN to serve pages from locations all over the world), then click "Create".

    (UPDATE: Member @thehound has found that bucket location DOES make a difference - although I still don't understand why it would, please feel free to experiment to see if you can improve your loading speed. Here are the relevant posts:

    https://stmforum.com/forum/showthrea...l=1#post371475

    https://stmforum.com/forum/showthrea...l=1#post371776

    You can just continue with the tutorial as-is for now, and experiment afterwards.)

    Note: It doesn't much matter what this bucket name is, as long as you remember which domain it's for. However, the name DOES need to be unique among all the buckets on Amazon (from all users and not just you). For my particular domain, I named my bucket "extremedeals.club".


    -Click on the bucket name you've just created. Click on the "Permissions" tab. "Block public access" should be highlighted by default (if not, select it). Click "Edit". UNCHECK all 4 options, and click "Save".



    Type "confirm" and click "Confirm":




    -Click on "Bucket Policy", and paste the following code into the Bucket policy editor, changing "MYBUCKET" in the code to the name of your S3 Bucket (don't forget this step!!):

    Code:
    {
  "Version":"2012-10-17",
  "Statement":[{
	"Sid":"PublicReadGetObject",
        "Effect":"Allow",
	  "Principal": "*",
      "Action":["s3:GetObject"],
      "Resource":["arn:aws:s3:::MYBUCKET/*"
      ]
    }
  ]
}
    Then hit "Save".




    -Click on the "Overview" tab. Click on "+ Create folder", type in the name "sweeps" (or whichever vertical you've chosen) and click "Save" (just leave "None" selected as by default).


    -On your desktop, open any text file editor (notepad would be fine for example), and write a line of text similar to the following, then save it as index.html:

    Nothing to see here!
    You can also save it as index.txt, and then change the file name to index.html.


    -Back in Amazon S3, click on "Upload" and then "Add files". On your computer, select the "index.html" file you've just created. Click "Next".


    -In the next step ("Set permissions"), under "Manage public permissions", change it to "Grant public read access to this object(s)". Then click "Upload" (no need to click through to "Set properties" and "Review" tabs - nothing to change there).

    Please note: Please follow this step for all files you'll be uploading to this bucket in the future.


    -Click on the "sweeps" folder you created earlier. Click on "+ Create folder", type in the name of the first prize of the sweeps offers you're planning to test - e.g. "iphonex" (no spaces) - and click "Save" (again, leaving "None" selected as by default).


    -Click on the "iphonex" folder, then click on "+ Create folder", and type in the language you've chosen to target first. You can type in the whole word or the 2-letter code for the language. Let's say you're wanting to run offers in Kenya, so the target language would be English (you can look it up on this page). In this case you can type in the folder name as either "english" or "en" (you can look up 2-letter country codes on this page). Click "Save" (again, leaving "None" selected as by default).


    -Download the sample landing page below - you can choose from .zip or .7z depending on which compression software you use (if you don't have winzip, download 7 zip from here):

    https://www.dropbox.com/s/s9l3p2ssmf4oa17/LP1.7z?dl=0

    https://www.dropbox.com/s/h50a9peaz91mylu/LP1.zip?dl=0

    Unzip the downloaded file.


    -Back in Amazon S3, click on the "en" folder, then click on "Upload", then drag the unzipped LP1 folder to the browser window, and click "Upload".


    -In the next step ("Set permissions"), under "Manage public permissions", change it to "Grant public read access to this object(s)". Then click "Upload" (no need to click through to "Set properties" and "Review" tabs - nothing to change there).


    -Click on the folder "LP1", then click on the file "index.html". Right-click on the link in the "Object URL" section at the bottom and choose "Open link in new tab" (or if you're not using the chrome browser, do whatever you need to do to open that link in a new browser tab).

    You should now see this landing page:



    If you do, it means you've set up the bucket policy and permissions correctly, so that the landing page is now accessible. Browsing to the landing page to make sure it is accessible, would always be a good check for every lander you upload in the future (because it you can't access it, then chances are your visitors won't be able to either).



    3)SET UP CDN on AMAZON CLOUDFRONT


    -In your Amazon AWS account, click on "Services" at the top, then go to "CloudFront" under "Networking & Content Delivery".

    -Click the "Create Distribution" button.

    -For "Select a delivery method for your content", we want to choose Web, so click on the "Get Started" button in the "Web" section.

    -You'll be in the "Create Distribution" screen. Click in the "Origin Domain Name" field and choose the S3 Bucket you've created above.

    -For "Default Root Object" enter "index.html".

    -Then scroll to the bottom and click "Create Distribution".

    You'll then see a screen like this:



    Copy the link in the "Domain Name" column and save it somewhere - we'll be needing it in the next step.

    (The "In Progress" will change to "Deployed" in around 15 minutes - no need to wait for this to happen before continuing with the next step.)



    4)SET UP DNS on ROUTE 53


    In the previous version of this tutorial, we were using NS1.com as the DNS. They weren't the cheapest to start with, and prices have increased between then and now (from $5 to $8 per million queries!) so I decided to go with Route 53, which is a LOT more cost-efficient, although there IS a compromise in redirection speed (around 60 milliseconds' difference at the time of writing). I'll compare the two services in greater detail in the EXPLANATION section.


    -In the AWS Console, click on "Services", scroll down to "Networking & Content Delivery" and click on "Route 53".




    -Under "DNS management" click on "Get started now". Click on "Create Hosted Zone".

    -Click on "Create Hosted Zone" (another one!) Fill in the domain name you registered from before (without the http://), and click "Create".




    -Click "Create Record Set". In "Name" type "www". Leave "Type" at the default "A = IPv4 address". For "Alias:" select "Yes". Click on the "Alias Target:" field and paste your cloudfront domain in here - again, you can find this in cloudfront:



    If, when you click on the "Alisa Target:" field, you see a list appear that shows your cloudfront domain, then just select it - no need for pasting. (I'm covering both situations - if your cloudfront distribution hasn't finished deploying, then you won't see the cloudfront domain in this list. Again, if you see it, select it; if you don't, paste it in.)

    Then click "Create".




    -Click "Create Record Set" again. In "Name" type "www". For "Type" select "AAAA = IPv6 address". For "Alias:" select "Yes". Click on the "Alias Target:" field and paste your cloudfront domain in here. Click "Create".




    -Next, we want to repeat the last 2 steps for the "naked" version of your domain, i.e. the one without "www". Click "Create Record Set" again. Leave "Name" empty. Leave "Type" at the default "A = IPv4 address". For "Alias:" select "Yes". Click on the "Alias Target:" field and paste your cloudfront domain in here. Then click "Create".


    -Click "Create Record Set" again. Leave "Name" empty. For "Type" select "AAAA = IPv6 address". For "Alias:" select "Yes". Click on the "Alias Target:" field and paste your cloudfront domain in here. Click "Create".


    -Lastly, we need to point our domain at Route 53's nameservers. If you look to the left, you'll see 4 entries for "NS":



    We need to log into Namecheap to enter these.


    -Log into namecheap, click "Domain List", find your domain and click on "Manage". On the "Domain" tab > "NAMESERVERS" section > click to change from "Namecheap BasicDNS" to "Custom DNS".




    -Click on "ADD NAMESERVER" twice to add 2 additional entries so there would be 4 in total. Then paste those 4 nameservers from Route 53 that we were looking at earlier. Don't forget to click on the little green checkmark to save.




    -Lastly, you'll likely need to wait for DNS to propagate - which will typically take less than 30 minutes.



    5)REQUEST FOR FREE SSL CERTIFICATE


    -Go to https://aws.amazon.com/certificate-manager/. (Or: In your Amazon AWS account, click on "Services" at the top, then go to "Certificate Manager" under "Security, Identity, & Compliance".)


    -Under "Provision certificates" click "Get started".


    -"Request a public certificate" should be selected by default. (If not, select it.) Click "Request Certificate".


    -In the "Add domain names" section, in the "Domain name*" field, put your domain name without "http://www.". So in my case, it would be "extremedeals.club".


    -Click "Add another name to this certificate" to add another blank field, then put into it your domain name preceded by "*.". So in my case I would put "*.extremedeals.club".




    -For "Select validation method", the option "DNS validation" should already be selected (if not, select it). Click "Review". Then click "Confirm and Request".


    -You'll then see that the "Validation status" of both versions of your domain will say "Pending validation". Click on the little black arrow/chevron in front of either one of the domains to expand the section. Click "Create Record in Route 53". Then click "Create". If all goes well you should see a "Success" message.




    -Repeat the previous step for the other domain version (i.e. expand the section, create record in route 53, then "Create").


    -You can refresh the page from time to time until "Pending validation" turn into "Success" and "Issued". Typically this will take less than 30 minutes. Once this has happened, proceed to the next step.



    6)ADD SSL CERTIFICATE TO CLOUDFRONT


    (Huge thank you to @edgekao who showed us this method in this post!)


    -Back in Cloudfront, click to choose the cloudfront distribution for your site, then click "General" -> "Edit".


    -In "Alternate Domain Names (CNAMEs)", type in there both the "www" version of your domain name and the version without "www" - both without "http://":




    -For "SSL Certificate", select "Custom SSL Certificate (example.com)". If this option is greyed out, it means that either:

    a)The certificate validation process has not finished - in which case wait for it to finish first, then try again.

    OR

    b)Try the suggestion in this post (thanks @zenion!): https://stmforum.com/forum/showthrea...l=1#post372618

    Once you've selected "Custom SSL Certificate (example.com)", click inside the empty field, and find and click on the new certificate that just got validated. Scroll down and click on "Yes, Edit" to save changes.



    7)LASTLY...


    -How do you know whether DNS has finished propagating? There are 2 things you can do. Firstly, you can check dnschecker.org. You can check both the "www" version and the one without, and if you see all green checkmarkss then you're OK. If you see red crosses, maybe wait some more before checking again. If after a few hours you're still not seeing ANY green checkmarks, then it may be time to go over all the instructions above to make sure you've set up everything correctly.




    -Another way to see if DNS has propagated, would be to simply browse to the sample landing page via your real domain, to see whether you can see the landing page. So, browse to:

    http://your-domain-name.whatever/swe...LP1/index.html

    In my particular case I would browse to:

    http://extremedeals.club/sweeps/ipho...LP1/index.html

    If you can see your landing page, then you're all set! If you can't, or get an error of some sort, then it may be because either DNS hasn't finished propagating, or you've made a mistake in the setup.


    -Also: If you can't seem to make things "work" even after verifying all the settings are correct, it may be because you've been changing some of the settings, but CDN is still serving up cached pages that were generated based on old settings. What you can try is go to Cloudfront, click on your distribution, click on the "Invalidations" tab, click "Create Invalidation", put a "*" in the "Object Paths" field, and click "Invalidate". When the "Status" goes from "In Progress" to "Completed", check to see if you can access your landers in the browser again.

    EXPLANATION

    As a non-geek that only knows just enough technical stuff to get by, I'm really in no position to explain how any of this works. However, I will briefly describe, in my rudimentary way, how the 4 components we've set up, work together to serve our landing pages. You can always learn more about this topic by googling.



    1)How the Domain, Amazon S3, Amazon Cloudfront, and Amazon Route 53 Work Together

    First of all, in order for files to be accessible on the internet, they must be hosted (stored) online somewhere.

    Amazon S3 is such type of storage space. So we upload our landers to it to be stored online and accessed by our visitors.

    Now - these landers are actually stored in a server which is located at a single physical location. Remember how S3 allowed for us to select a "Region"? That would be the location the pages would be served from.

    Trouble is, because we'll probably be promoting offers from all over the world, some of those target geos may be very far from the server location - and the further the visitor is from the server, the longer they'll have to wait for the landing page to load.

    It is extremely essential for us to make our landing pages load quickly, because the slower a page loads, the more visitors will become impatient and leave the page. According to Caurmen's testing, "2 seconds of loading time can equate to more than a 100% increase or decrease in CTR."

    One of the best ways to make landers load faster, would be to use a CDN service. What it does is it will send copies of the lander to multiple server locations around the world, so that when a visitor tries to access your lander, the CDN will serve the page from the server nearest that visitor. This is where Amazon Cloudfront comes in - it's a CDN service that will grab the files you upload to S3 and distribute them to multiple servers around the world.

    Next, remember what the S3 and Cloudfront links looked like? They look something like the following...

    Sample S3 link: http://s3.amazonaws.com/extremedeals...LP1/index.html

    Sample Cloudfront link: http://d1ugld9y4dv05s.cloudfront.net...LP1/index.html

    You'll probably admit that these links don't look very relevant to the offers we're wanting to run, so may cause some confusion for our visitors. This was why we needed to register a relevant Domain Name.

    Lastly, we need something that will link the domain name to the files on the CDN. This is the purpose of Route 53.

    Remember that in the domain settings on Namecheap, we specified the nameservers as the ones we got from Route 53? This way, when a visitor tries to access the domain name in the browser, they will be sent according to the settings at Route 53 - which will redirect the visitors to the CDN domain.

    i.e. On Route 53, we've set up ALIAS records for the both extremedeals.club and www.extremedeals.club, that point these domain versions to the CDN domain d1ugld9y4dv05s.cloudfront.net. So if anyone tries to browse to for example:

    http://extremedeals.club/sweeps/ipho...LP1/index.html

    Or:

    http://www.extremedeals.club/sweeps/...LP1/index.html

    Route 53 would know to direct them to:

    http://d1ugld9y4dv05s.cloudfront.net...LP1/index.html



    2)Is Route 53 Fast Enough?

    In the previous version of the tutorial, we used NS1.com as the nameserver provider instead of Route 53. Here I want to do a more details comparison, to give you the option to switch to NS1 if you see fit.

    The first consideration is redirection speed. In this aspect, NS1 beats Route 53 by around 60ms at the time of writing:

    https://www.dnsperf.com/

    How much impact this difference will have on campaign performance is unknown - unless we were to run a split-test to compare conversion rate. This would be an interesting exercise which I don't have time for at the moment. But please feel free to run your own tests to find out!

    The other consideration is pricing. NS1 has a free plan where the first 500k queries of each month are free, and additional queries will cost $8 per million queries. Compared to the $0.40 per million queries charged by Route 53, the difference is considerable. Having said that: If you're running high volume traffic where you'd be hit with a big bill from NS1, you'd hopefully be making enough profits so that this expense would be small by comparison. But I decided that it would be nice to help newbies to save on business costs if possible.

    Related: Member Rubik has alerted us to a potential issue with NS1:

    https://stmforum.com/forum/showthrea...l=1#post335123

    I've been using NS1 for several years now without issues, but I'll leave you to make the decision which DNS service to go with.

Instead, you'd need to either overwrite the S3 file with an updated one, or download the file from S3 to desktop, edit it, and re-upload it. Whereas if you were hosting the files on a server, you'd be able to edit the files on the server directly (this is actually not 100% accurate, but it would SEEM that way, as sublime text would handle the downloading and uploading and rewriting - don't worry if you don't know what I'm talking about at this point, as I'll refer back to here in the next lesson).

    2). If a domain goes down and needs to be replaced, you NEED to have one ready to replace it and not just set one up then and there (otherwise the first day performance is going to suck)

    3)Setting up meaningful subdomains can potentially increase your conversion rate:

    ...as reported by member @mywebexperts, who has kindly describe how to implement subdomains here:

    https://stmforum.com/forum/showthrea...l=1#post386324


    4)Last thing I want to tell you: Advertizer.com had reported that https links converted 4x better than http links:

    https://stmforum.com/forum/showthrea...ntiom-to-tip-2

    But matuloo and some of the other members reported no positive impact on conversion rates:

    https://stmforum.com/forum/showthrea...sl-help-ot-not

    If you'd like to implement https on Route 53 and Cloudfront, edgekaos wrote an excellent tutorial:

    I just had this issue when setting up a new domain for myself. I don't recall having this problem when doing the same thing few months ago (and they still allow that domain to work :P )

    If anyone is using Namecheap, I chose to request certificate validation via DNS. This means you gotta go to namecheap user panel and in the Advanced DNS section, choose to add a new record, following this format:

    Type: CNAME record
    Host: (AWS)Name and only include until www without . , so if you have _cff0cda88701846cbe7a34cd737378e2.www.example.com, do it as _cff0cda88701846cbe7a34cd737378e2.www
    Value: (AWS)Value (the whole thing)
    TTL: leave it on default setting

    And make sure to give it at least 1-2 hours for it to update and confirm certificate issue.
